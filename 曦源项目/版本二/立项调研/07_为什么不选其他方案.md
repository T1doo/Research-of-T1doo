# 07 · 为什么不选其他方案

> **目的**：把 6 候选方案 + 候选 B 不入选的理由清晰记录在案，避免 1 个月后忘记，也方便导师审阅时秒懂决策依据。

## 1. 排除候选 B · FocusVLA-Robust-Mini（架构改造路线）

### 候选 B 是什么
手写 Modality Cascaded Attention（FocusVLA 核心组件）接入 SmolVLA-0.5B + 加 TRADES-mini 一致性损失，验证"架构鲁棒 × 数据鲁棒"的叠加增益。

### 为什么 Plan agent 不推荐

1. **工程风险显著高于科研风险**：FocusVLA 公式 1-5 写得简洁，但 attention 模块的实际实现涉及：
   - KV cache 重新规划
   - causal mask 在 3 路串行 attention 间的传递
   - 与 LoRA 的兼容（LoRA 默认装在 q_proj / v_proj 上，串行后 q_proj 共享 vs 独立的选择）
   - 梯度回传的稳定性
   这些坑对 2 名大二本科生在 1-2 个月内调通的把握 < 30%
2. **代码未释 + 实现 ambiguity**：FocusVLA 论文没披露超参（如 fusion 函数 σ_fusion 的具体形式），即使写出来也无法验证是否是论文原版
3. **没有 fallback**：候选 B 的核心动作就是架构改造，如果失败只能整个倒回候选 A 路线，时间损失 3-4 个月
4. **Clean SR 退化风险**：架构改造后即使能训，clean baseline 可能掉 5-10pp，把鲁棒性增益完全吃掉

### 但候选 B 不是完全无价值
候选 B 适合作为 **Plan A 的 stretch goal**：若 P1-P3 完成快、团队 PyTorch 能力远超预期，可以在 M7-M9 尝试用 Cascaded Attention mini 替换 OpenVLA-OFT 默认架构，作为附加贡献。
**但这是 nice-to-have，不在主路径里。**

## 2. 排除原综述候选 3 · TT-VLA-Pro（Test-Time Adaptation）

### 原方案是什么
Test-Time Adaptation：用 VLM verifier (InternVL3 / GPT-4V) 作 task progress reward，加 RC-NF safety constraint，在 VLA-Risk 6 维上做在线适应。综述报告排序第一（4.25/5）。

### 为什么不选

1. **依赖 GPT-4V / InternVL3 等大 VLM API**——对 ¥10K 预算极不友好。GPT-4V 每次 inference 约 ¥0.5，TT-VLA 一个 episode 可能调用 30-50 次，500 episode 评测就要 ¥7500-12500，**单评测预算耗尽**
2. **不"围绕 FocusVLA"**——这是 C 主线（补盲方向），与导师推荐的 A 主线（视觉利用）不对齐
3. **VLA-Risk 代码未释**，主评测平台不稳定
4. **TT-VLA (2601.06748) 原论文的"secret sauce"是手工设计的 reward**，替换为 VLM verifier 是一次重设计，风险高

### 何时可以回头看
若团队在 M6-M7 已基本完成 Plan A 主体，且发现 InSpire failure predictor (H3) 在 attention map 上无效——可以考虑把 P4 替换为 TT-VLA 风格的在线调整探索。但这是末期 stretch。

## 3. 排除原综述候选 2 · Recovery-Aware VLA

### 原方案是什么
RobustVLA 主推策略 + RC-NF 失败检测 + 主动恢复闭环。综述报告排序 3.875/5。

### 为什么不选

1. **需要 RobustVLA 全套训练**——单 LoRA r=32 + worst-case δ 训练显存约 50-60GB，超出 46GB 单卡上限
2. **需要采集 recovery trajectory 数据**（综述报告说"1-2 人月可采 10k"，但 2 大二本科生无真机经验，估计 3-4 人月）
3. **不"围绕 FocusVLA"**——这是 B+C 交叉，与 A 主线不对齐
4. **RC-NF 代码访问难**（论文是 References/ 已读，但不开源）

## 4. 排除原综述候选 4 · Instruction-Robust VLA

### 原方案是什么
RobustVLA 框架扩展到 instruction 维度，PAIR / GCG 生成对抗指令 + 正确 action 配对做 LoRA 微调。综述报告排序 3.875/5。

### 为什么不选

1. **PAIR / GCG 是 NLP 攻击算法**——对 2 大二本科生（机器人方向）来说，跨领域门槛过高
2. **需要 LLM-based 对抗指令生成**，又涉及 API 预算（GPT/Claude API 累计估 ¥5000+）
3. **方向偏 NLP 而非 visual robustness**，与 FocusVLA 锚点偏离
4. **数据生成质量难控**——对抗指令的"语义保真"很难自动验证

### 但 instruction 维度的评测仍保留
Plan A 的 7 维 LIBERO-Plus 评测包含 instruction 维度；如果发现该维度 InSpire / Focus 都不 work，可以在 §Discussion 中引用本路线作为 future work。

## 5. 排除原综述候选 6 · Adversarial RoboGen

### 原方案是什么
用 LLM 在 RoboGen / MimicGen 上生成对抗指令 + 配对 action，作为训练增广。综述报告排序 3.75/5。

### 为什么不选

1. **完全新方向，没有先例可对照**——综述报告也说"风险高"
2. **需要协调 RoboGen + adversarial prompting + RobustVLA 三个 pipeline**——工程量是 Plan A 的 2-3 倍
3. **LLM API 成本**：合成数据生成估计需要 ¥5000+ API 预算
4. **不"围绕 FocusVLA"**——是 B+C 主线交叉

## 6. 排除原综述候选 5 · Shortcut Learning 诊断与防御

### 原方案是什么
[[2025-08_ShortcutLearning_Xing]] + DRO + CAST/CF-VLA 的"诊断-架构-数据"三方组合。综述报告排序 3.625/5（高赌注）。

### 为什么不选

1. **诊断阶段实验设计极复杂**——需要量化 shortcut（"左手拿杯 = 蓝色背景"这类 spurious correlation），工程方法不成熟
2. **依赖多个论文同时复现**——CAST / CF-VLA / ShortcutLearning toolkit 都未释或仅部分释
3. **故事拍不响的风险大**——综述自己评 "如果没找到强 shortcut 故事拍不响"
4. **本科生承受度低**——需要同时理解 DRO 数学 + counterfactual augmentation + shortcut detection 三套工具

### 但 ShortcutLearning 的诊断思想已融入 Plan A
Plan A 的 §Discussion 会引用 [[2025-08_ShortcutLearning_Xing]] 作为「为什么需要显式 grounding」的理论依据，相当于"小尺度沿用"。

## 8. 关键决策原则（写给未来的自己）

1. **导师推荐 = 强约束**——"围绕 FocusVLA" 是非否决项
2. **2 人本科团队 = 工程量约束** ——任何需要重写 transformer 内部模块的方案都要慎选
3. **¥10K = 预算约束**——任何依赖大 LLM API 累计 ¥5K+ 的方案不可行
4. **1 年 = 时间约束**——任何需要真机的方案不可行
5. **想投论文 = 故事完整性约束**——必须有明确的 H 假设 + 完整对照实验

## 9. 如果 1 个月后想反悔...

如果在 W1-W2 期间发现 Plan A 比预想更轻松（如方向词 GT 一次跑通），可以**有条件升级**：

- **轻升级**（推荐）：把 Plan A scope 从 LIBERO-Plus 7 维扩展到 LIBERO-Plus + LIBERO-Para（语言改写）+ VLA-Risk 子集
- **中升级**：在 P5 末（M11-M12）尝试嫁接 RobustVLA TRADES-mini consistency loss（候选 B 的精简版）
- **重升级**（不推荐）：切换到候选 3 TT-VLA——这会重做整个方案，损失 P1-P2 的 3 个月工作
