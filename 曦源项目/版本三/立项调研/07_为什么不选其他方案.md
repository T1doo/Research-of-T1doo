# 07 · 为什么不选其他方案

> 本文档记录版本三立项过程中，**对每一条被考虑过但最终未选**的研究路线的决策依据。便于未来回顾 / 给审稿人解释 / 给老师讨论。

## 1. 不延续版本二（InSpire 显式提示插件 + 视觉利用 ablation）

**未选原因**：

1. **用户明确指示版本三是"全新路线"**——不是版本二的延续或并行候选。
2. **版本二依赖训练 → 与"纯评测"约束直接冲突**：方向词 GT 标注 + cls head + LoRA loss 联合训练，工程量大且训练侧风险（GT 噪声 / cls 收敛 / loss weight 调参）已经在 v2 修订中显现。
3. **版本二 framing 隐性不严谨**：v2 修订前曾用 OpenVLA-OFT 默认架构作 "Focus-style 代理"，被外部审阅指出不严谨。版本三完全规避这类 framing 问题（不做 "代理"，直接用 SAM 当外部 oracle）。
4. **导师建议"在 FocusVLA 方向深挖鲁棒性"**——版本三直接回应 FocusVLA 自述局限 #3 #4，叙事比版本二（间接通过 InSpire 显式 grounding）更紧密。

**版本二的价值**：作为**未来工作 / appendix** 仍可保留——版本三的 H1 / H2 验证成立后，可以进一步说"如果想训练 VLA 提升鲁棒性，可以加 InSpire 提示插件"。两条路线在长期是互补的。

## 2. 不复现 / 改造 FocusVLA 架构

**未选原因**：

1. **工程超纲**——Modality Cascaded Attention 需要重写 transformer attention block；Focus Attention 涉及 KV cache / TopK 可微化 / channel gate 等多个 PyTorch 工程坑。
2. **FocusVLA 代码未释**（截至 2026-05-26）——abstract 承诺"will be made public"但 GitHub 还没看到。
3. **与"纯评测"约束直接冲突**——重写架构 + 重训不可能不动 VLA。
4. **被版本二 v1 在 W2 末 Go/No-Go 评估时已经排除**（见 [[版本二/立项调研/07_为什么不选其他方案]]）。

**判断**：FocusVLA 复现这条路是博士生 6 个月级别的工程，不是大二本科生 1 年期项目能完成的。

## 3. 不复现 Oat-VLA / SlotVLA / TokenFLEX 等变长 / object-centric VLA

**未选原因**：

1. **都需要大量训练**：Oat-VLA 需要训 object-agent pooling 模块；SlotVLA 需要训 slot attention；TokenFLEX 需要训练时随机 K。**与"纯评测"约束直接冲突**。
2. **novelty 已被占**：作为"新方法"做工程开创的空间已经被这 3 篇论文 + Compressor-VLA + SemanticVLA + VLA-Pruner + LUVC 等 ~10 篇 token pruning 工作占领。
3. **复现工程量超 1 年期**：每个的官方训练 recipe 通常需要 ≥ 100 GPU-h 训练 + 50 GPU-h 评测，超本项目预算。
4. **2 大二本科生不具备 from-scratch 实现这类架构的能力**——这是博士生级别工程。

**判断**：与其在新方法的红海里厮杀，不如转向"评测/诊断"蓝海。版本三 SAM-as-Probe 是这种转向。

## 4. 不做"SAM mask 作为弱监督训练 VLA"

**未选原因**：

1. **需要重训 VLA → 违反"纯评测"约束**。
2. **审稿人会问"为什么不直接做端到端 SAM-VLA 联合训"** —— 这种端到端方案工程门槛极高（SAM 是 frozen 还是 unfrozen？mask loss 与 BC loss 如何 balance？），开放问题太多。
3. **GuidedVLA (arXiv:2605.12369, RSS 2026) 已经做了相邻的工作**（用 SAM/GroundingDINO 自动标注 object bbox 作 supervisory signal），如果做这条路线会撞车。

**判断**：把 SAM 当**外部 oracle 来评测**，比当**内部弱监督来训练**更轻量、更 novel、更安全。

## 5. 不直接做"变长 token 的 VLA 微调"

**未选原因**：

1. **需要重训 VLA**。
2. **TokenFLEX (arXiv:2504.03154) 已经抢先**——训练时随机 K 实现真正变长。
3. **本项目通过 H2（K\* 分布画像）已经能从评测侧给出"为什么应该变长" 的实证证据**，不需要自己训练一个变长 VLA 来验证。
4. **One Token Per Frame (arXiv:2605.07931)** 实现了更极端的变长（每帧 1 token），novelty 空间被进一步压缩。

**判断**：variable-K 本身已被做。本项目通过 K\* 分布画像在"评测侧"做贡献，比自己训一个新的变长 VLA 更可行且更 novel。

## 6. 不做"在 LIBERO-Plus 上系统评测 token pruning 工作"

**未选原因**：

1. **太被动**——纯 benchmarking 6 个 token pruning 工作，故事缺乏 anchor，难以投出。
2. **与版本二 Plan B（FocusVLA 鲁棒性多维诊断）方向重叠**——版本二 Plan B 已经覆盖类似评测。
3. **缺少外部参考真值** —— "我们评测 VLA-Pruner 和 LUVC 哪个在扰动下更鲁棒" 没有 anchor，难以给出可操作的结论。

**判断**：版本三引入 SAM mask 作外部 oracle，把"评测"升级为"诊断"——这是关键 framing 升级。

## 7. 不做"真机 SAM-as-Probe 评测"

**未选原因**：

1. **2 大二本科生没有真机访问权**（团队预算 ¥10K 也无法租真机）。
2. **真机 episode 收集成本极高**（每 episode ~5 min × 100 episode × 多次重复 ≈ 50h）。
3. **真机 attention dump 受相机 / 控制器延迟影响**，方法学问题更多。

**判断**：仿真先行（LIBERO + LIBERO-Plus），真机作为未来工作。这与导师叙事一致——FocusVLA / Oat-VLA / TokenFLEX 等工作主体都是仿真。

## 8. 不做"加入 VLA-Risk 6 维"作为主战场

**未选原因**：

1. **VLA-Risk 代码 + 数据未释**（截至 2026-05-26）。
2. **VLA-Risk 论文已提交 ICLR 2026 但仍 anonymous**，访问需要邮件作者。
3. **本项目的 H1-H3 在 LIBERO-Plus 7 维已经能成立**，加 VLA-Risk 是 stretch 而非必要。

**判断**：VLA-Risk 作 stretch goal（M5 末若有 GPU 余量 + 代码 release）。

## 9. 不做"加入 LIBERO-PRO / LIBERO-X / Colosseum 等其他 benchmark"

**未选原因**：

1. **LIBERO-Plus 7 维已足够覆盖主要扰动类型**——添加更多 benchmark 不增加 H1-H3 验证力，反而稀释 GPU 预算。
2. **LIBERO-PRO 更激进**（SR 90% → 0%），但与 LIBERO-Plus 同质化高（都是 LIBERO 扰动版）。
3. **Colosseum / RoboCasa / SimplerEnv 等是不同仿真平台**，跨平台对照需要额外仿真器装载工程。

**判断**：聚焦 LIBERO-Plus 主战场，避免 benchmark 过载。

## 10. 不做"细粒度 keypoint mask 替代物体 mask"

**未选原因**：

1. **作为主路线工程复杂度过高**——keypoint 选择本身需要额外模型（手 / 物体接触点检测器）。
2. **物体级 mask 已经能验证 H1-H3** —— 不必先用 keypoint。

**判断**：keypoint mask 作为 P4 失败模式画像的**补强手段**（仅在 H1 部分否定时启用），不在主路线。

## 11. 总结：本项目的方法学选择哲学

| 维度 | 我们的选择 | 为什么 |
|---|---|---|
| 训练 vs 评测 | **评测** | 2 本科生 1 年期 + ¥10K 不适合大规模训练；user 明确约束 |
| 新方法 vs 新评测 | **新评测/诊断** | "新方法" 的红海已被 ~30 篇 VLA 工作填满；"新评测" 蓝海空白明确 |
| 外部 oracle vs 内部 attention | **外部 oracle（SAM mask）** | 内部 attention 评内部 attention 是循环论证；外部 oracle 是首次 |
| 仿真 vs 真机 | **仿真（LIBERO-Plus）** | 真机不可得；仿真足够支撑 H1-H3 |
| 一个深度 benchmark vs 多个 benchmark | **一个深度（LIBERO-Plus 7 维）** | 多 benchmark 稀释预算 |
| 单 backbone vs 多 backbone | **2 backbone（OpenVLA-OFT + SmolVLA）** | 2 个足够形成 14 数据点验证 Pearson 相关；3 个超预算 |
| 物体 mask vs keypoint mask | **物体 mask 主线 + keypoint stretch** | 物体 mask 工程简单且足够 |
