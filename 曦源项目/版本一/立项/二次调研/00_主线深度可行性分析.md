# 主线深度可行性分析

> [!info] 主线最终定义
> **「以 HuggingFace lerobot SmolVLA (0.45 B) 为统一骨干，注入 AAC 推理时 entropy-driven chunk 调度 + RobustVLA 训练时 input/output 双正则化（LoRA r=32 包裹），形成 Capacity-Efficient Multi-Modal Robust VLA」**。本文给出该主线在训练显存 / 推理延迟 / 训练-推理一致性 / 创新点 / 风险 五个维度的二次定量分析。

## 1. 训练显存账本（精算到 GB）

### 1.1 单独构件显存（batchsize=2, action chunk H=50, AdamW, bf16）

| 构件 | 参数量 | Forward 激活 | Optimizer state | 合计（训练态） |
|------|--------|------------|----------------|--------------|
| SmolVLA 全参微调（baseline） | 0.45 B (350M VLM + 100M expert) | ~2 GB | ~1.8 GB (param×4×bf16) | **~6 GB** |
| SmolVLA + LoRA r=16（仅 expert+VLM attn QKV） | 可训 ~10M | ~2 GB | ~40 MB | **~4 GB** |
| SmolVLA + LoRA r=32（agent 2 建议升级） | 可训 ~20M | ~2 GB | ~80 MB | **~4.2 GB** |
| π0 全参微调（对照） | 3.3 B | ~12 GB | ~13 GB | **~46 GB**（立项基线） |
| π0 + LoRA r=16 | 3.3 B + ~50M 可训 | ~12 GB | ~200 MB | **~16 GB** |

### 1.2 + AAC 推理时 N=20 采样的显存追加

AAC 仅作用于推理，**训练阶段无显存增量**；推理时 N 次 forward 复用 KV cache，激活 peak ≈ 单次 forward × N，但因为 batch=1 推理，绝对值仍小：
- SmolVLA + AAC N=20 推理：~3.5 GB（远低于训练态）
- AAC N=5 推理（快速档）：~2 GB

### 1.3 + RobustVLA 双正则化训练的显存追加

这是显存膨胀的主战场。RobustVLA 三项改动各自显存增量：

| 改动 | 公式简述 | 显存增量（vs SmolVLA baseline 6 GB）| 备注 |
|------|---------|-----------------------------------|------|
| **Output-robust（inner max 找最坏 δ）** | $\max_\delta \|v_\theta - u - \delta\|^2$ | **+5～8 GB**（PGD-3 步 + 二阶图） | 主要负担来自二次反向 |
| **Input-robust（worst-case 输入扰动）** | $\max_{\omega^*} \|v_\theta(o_\omega) - u\|^2$ | **+3～5 GB** | 视觉/语言扰动产生额外 forward |
| **UCB 多臂老虎机（17 arms 选 worst）** | 增 logit/reward 状态向量 | < 1 GB | 几乎可忽略 |

**朴素叠加**：SmolVLA + 全参 + RobustVLA ≈ 6 + 5 + 4 + 1 = **~16 GB**（仍在 24 GB 单卡可行）

### 1.4 三档 GPU 可行性对照表（最关键账本）

| GPU 档位 | SmolVLA full | + RobustVLA full | + LoRA r=32 替代 full | + AAC 推理 N=20 |
|---------|-------------|-----------------|-------------------|----------------|
| **RTX 4090 (24 GB)** | ✅ 18 GB 余量 | ⚠️ 8 GB 余量，可跑但脆弱 | ✅ ~10 GB，舒适 | ✅ ~3.5 GB（推理独立） |
| **A6000 / A100 40 GB** | ✅ 大量余量 | ✅ ~24 GB 余量 | ✅ ~36 GB 余量 | ✅ |
| **A100 80 GB（立项基线）** | ✅ 极宽裕 | ✅ ~64 GB 余量 | ✅ | ✅ |

**关键结论**：
- 在 24 GB 单卡 4090 上**全套主线（SmolVLA + RobustVLA + LoRA r=32 + AAC 推理）可跑**，但全参 RobustVLA 会让 4090 接近上限，推荐走 LoRA 路径
- 项目硬件预期（80 GB A100）下，所有方案均有 ≥ 50% 显存余量
- **Free-AT 单循环优化（Shafahi 2019, arxiv 1904.12843）** 是显存安全网：把 outer-min/inner-max 改为 m-step replay，可把 RobustVLA 显存增量从 +5～8 GB 压到 +0 GB（与 SFT 同档），如遇 OOM 立即启用

### 1.5 Agent 3 提出的 AAC × RobustVLA 联合显存峰值警告

Agent 3 警告"AAC N=20 重采样 + RobustVLA 双循环叠加 → 显存峰值 ×3"。**核账后澄清**：这不是训练态问题（AAC 训练时不启用 N=20），而是**联合 inference + 训练评估**时的峰值。我们在训练阶段只跑 RobustVLA，推理阶段只跑 AAC，二者时序错开 → **峰值不叠加**。仅在做"训练中周期性 eval"时需要谨慎管理（评估时关闭 RobustVLA 反向图、AAC N 调小即可）。

## 2. 推理延迟逐项叠加（毫秒级）

### 2.1 基线参考

| 配置 | 端到端延迟 | 控制频率上限 | 数据来源 |
|------|-----------|------------|---------|
| SmolVLA 同步 forward | ~50 ms | 20 Hz | 一次调研 Agent 1 |
| SmolVLA 异步（队列阈值 g=0.7） | 等效 ~30 ms 摊销 | 33 Hz | SmolVLA 论文 |
| π0 同步（参考） | 73 ms (14+32+27) | 13 Hz | 一次调研 Agent 2 |

### 2.2 主线方案延迟逐项加和

| 配置 | 单步延迟 | 控制频率 | 备注 |
|------|---------|---------|------|
| SmolVLA 同步 baseline | 50 ms | 20 Hz | — |
| + AAC N=5 推理 | 50 + ~15 = **65 ms** | 15 Hz | 边际成本（AAC 原论文给的是 +23 ms@N=20） |
| + AAC N=20 推理 | 50 + ~25 = **75 ms** | 13 Hz | — |
| + 异步推理（与 AAC 并行） | 摊销 **30~40 ms** | 25~33 Hz | AAC 截断作为提前触发，需精心设计 |
| **目标终态**（同步内核 + 异步调度 + AAC N=5 + RTC chunk 边界） | 摊销 **25~30 ms** | **30~40 Hz** | 论文卖点：从 SmolVLA 20 Hz 提升到 30+ Hz |

### 2.3 关键工程细节

- **RTC (Real-Time Chunking, Agent 1 发现的 arxiv 2506.07339)** 是 SmolVLA 异步 stack 缺失的关键组件，处理 chunk 边界跳变。**已在 π0.6 上 drop-in 集成验证**，应作为主线 baseline 之一。
- **AAC 截断 → async queue 提前触发**是本主线的 systems 创新点（详见 [[06_创新点定位与论文卖点]]）。在 SmolVLA 的 `RobotClient.receive_actions/_aggregate_action_queues/control_loop_action` 中已有 `must_go` 触发器（Agent 3 发现），可直接挂 prefetch hook。
- **A2A 1-step flow matching**（一次调研发现）作为 stretch goal：若集成后单次 forward 从 50 ms 降到 ~5 ms，整体可冲击 100 Hz 控制频率。

## 3. 训练-推理一致性问题

这是主线**最具理论深度的可行性问题**：RobustVLA 训练时让模型见过 worst-case action noise δ，AAC 推理时用 N 次采样估计动作分布的熵 — 这两个机制的训练-推理一致性如何？

### 3.1 三个具体子问题

1. **熵估计偏差**：RobustVLA 训练后的 flow matching velocity field 已经被"拉平"（worst-case δ 增加了模型对噪声的容忍），那么 AAC 用 N 次采样估的 $\Sigma_t$ 会不会**系统性偏小**（因为模型变得更"自信"）？
2. **触发阈值漂移**：AAC 的 chunk 截断阈值 $h^*=\arg\max_h(\bar E_{h+1}-\bar E_h)$ 在 vanilla SmolVLA 上调好，挂到 RobustVLA-trained SmolVLA 上是否需要重调？
3. **覆盖率假设**：RobustVLA 训练时见的是 17 类扰动；AAC 推理时遇到的扰动可能不在这 17 类里。模型对"未见扰动"的熵估计是否仍可靠？

### 3.2 主线立场（以实验回答而非假设）

我们**不**事先假设这三个问题无解。Phase 1（AAC@vanilla SmolVLA）和 Phase 2（RobustVLA@SmolVLA）分别独立训练 + 评测，**Phase 3 联合实验时必须做**以下消融才能证明一致性：

| 消融 | 目的 | 决策 |
|------|------|------|
| AAC@vanilla SmolVLA vs AAC@RobustVLA-SmolVLA 的 chunk 长度分布对比 | 检测 entropy 是否被 RobustVLA 训练拉平 | 若分布有显著漂移，重调 ξ 阈值 |
| AAC 在 17 个 in-distribution 扰动 + 5 个 OOD 扰动下的成功率 | 检测覆盖率假设 | 若 OOD 扰动下 AAC 失效，加入 calibration 模块 |
| 关闭 AAC / 关闭 RobustVLA / 全开 三种配置在 LIBERO-Plus 的成功率 | 检测两机制是否冲突 | 决定是否需要联合训练（而非独立训练后组合） |

### 3.3 理论联系（潜在 paper 卖点）

观察：**RobustVLA 的 worst-case δ 和 AAC 的 N 次采样高斯熵 $E_t = \frac{1}{2}\log[(2\pi e)^d \det(\Sigma_t)]$ 之间存在 dual relation**：
- δ 度量"最坏情况下模型预测能偏多远" → 上界 robustness margin
- $\Sigma_t$ 度量"自然采样下模型预测分布多宽" → 下界 epistemic uncertainty

如果能形式化证明 $\|δ^*\|^2 \propto \mathrm{tr}(\Sigma_t)$ 的 PAC-Bayes 关系，那 AAC × RobustVLA 就有了**统一理论框架** —— 这是 contribution 中 theoretical 层最值钱的卖点。即使不证明完整 bound，给出**实证 correlation 曲线**也足够发顶会。

## 4. 创新点初步定位（详见 [[06_创新点定位与论文卖点]]）

### 4.1 三层 contribution

| 层级 | 内容 | 与现有工作差异 |
|------|------|-------------|
| **Systems** | SmolVLA + AAC entropy 截断 + 异步 queue prefetch 的耦合调度算法 | AAC 原论文是同步 GR00T 实验，异步耦合是新的 |
| **Empirical** | < 1B VLA 上首次 17 多模态扰动 + LoRA + worst-case 双正则的端到端验证 | 文献 421 篇候选中无交集（Agent 2 核对） |
| **Theoretical** | worst-case δ 与 entropy uncertainty 的 dual relation（PAC-Bayes 风格 bound 或 empirical correlation） | 完全空白，即使弱版本也是 novelty |

### 4.2 主线 framing（采纳 Agent 4 修订建议）

**论文标题候选**：*"Capacity-Efficient Multi-Modal Robust VLA: Coupling Adaptive Inference with Worst-Case Regularization in 0.45 B Parameters"*

不再用一次调研的 "小模型鲁棒训练增益更大" 强 claim，改为更稳的 **"小模型在 worst-case 数据增广范式下能达到与大模型相当甚至更优的相对增益"**（条件成立判断来自 Agent 4 + 补强校验）。

## 5. 风险与缓解（二次细化）

### 5.1 高风险（red）

| 风险 | 来源 | 概率 | 缓解 |
|------|------|------|------|
| **RobustVLA 是 JAX/Flax 实现，需 1:1 翻译到 PyTorch** | Agent 3 代码侦察 | 高（确认事实） | Phase 1 末尾预留 1 周翻译 + 单元测试；找 openpi 仓库（社区有 PyTorch 端口）做 starting point |
| **LoRA r=16 在少样本下不够（Agent 2 警告）** | Few-Shot Adv LoRA (2505.15130) | 中高 | 直接采纳建议升级到 **r=32**；预算消融跑 r=8/16/32/64 看曲线 |
| **AAC 在 RobustVLA-trained model 上的熵估计偏差**（§3.1） | 理论推测 | 中 | Phase 3 联合实验必跑分布漂移消融 |
| **StreamingVLA (2603.28565) 主张抛弃 chunking 路线** | Agent 1 发现 | 中（被引用） | 论文 introduction 明确定位 chunking + streaming 是正交维度（chunking 是边界处理、streaming 是连续输出） |

### 5.2 中风险（yellow）

| 风险 | 来源 | 缓解 |
|------|------|------|
| **TT-RTC 已在 π0.6 drop-in 验证为社区默认升级** | Agent 1 发现 | 把 RTC + TT-RTC 作为 SmolVLA 基线必装件，AAC 是其正交补充（chunk 长度 vs chunk 边界） |
| **A2C2 在 SmolVLA+LIBERO d=8 已 >90%** | Async Comparison (2605.08168) | 改 framing 重点从"绝对性能"到"扰动鲁棒性 + 推理频率"双维度 |
| **PolicyServer 多线程调度复杂度** | Agent 3 代码分析 | Phase 1 先做单线程 prefetch baseline，多线程留 Phase 3 |
| **训练中显存峰值不可预测** | RobustVLA 双循环 | Free-AT (1904.12843) 单循环作 fallback；监控 nvidia-smi 设 +1 GB safety margin |

### 5.3 低风险（green）

- LIBERO benchmark：成熟，47× throughput（Allen AI Harness）；
- SmolVLA 异步 stack：已开源（HF lerobot），有完整 PolicyServer / RobotClient 实现；
- 8 个 AAC 注入点 + 8 个 RobustVLA 注入点已由 Agent 3 在 file:function 级标好（详见 [[03_SmolVLA代码可注入点分析]]）。

## 6. 可行性最终结论

**可行性评分：A-**（一次调研给的是 A，二次调研发现 2 个具体细化项后微调）：

- ✅ 显存账本可行（24 GB 4090 即可，80 GB A100 极宽裕）
- ✅ 推理延迟可控（目标 30+ Hz 控制频率有清晰路径）
- ✅ 主线无实质性抢做（Agent 2 在 421 篇候选中无交集）
- ✅ 代码注入点定位完整（16 个，Agent 3 file:function 级）
- ⚠️ JAX→PyTorch 翻译需 1 周（已纳入 Phase 1 时间表）
- ⚠️ LoRA r=32 而非 r=16（已采纳 Agent 2 建议）
- ⚠️ 训练-推理一致性需 Phase 3 消融验证（不是 blocker，是 contribution 的一部分）

**Go decision**：主线可推进 Phase 0 启动。具体步骤见 [[05_完整科研技术路线]]。
